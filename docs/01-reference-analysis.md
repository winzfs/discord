# 01. 참고 문서 분석 및 재해석 기준

## 1. 분석 목적

업로드된 참고 문서는 `운빨존많겜` 계열의 랜덤 협동 디펜스 장르 구조를 이해하기 위한 자료입니다.

이 문서의 목적은 다음과 같습니다.

1. 장르적으로 어떤 재미가 작동하는지 정리한다.
2. 어떤 시스템을 우리 프로젝트에 재해석할 수 있는지 분류한다.
3. 저작권과 서비스 정체성 문제를 피하기 위한 금지 기준을 명확히 한다.
4. 디스코드 커뮤니티 웹게임에 맞게 축소/변형할 지점을 정한다.

## 2. 참고 장르의 핵심 구조

참고 자료에서 확인되는 장르적 구조는 다음과 같습니다.

### 2.1 랜덤 소환

플레이어는 자원을 사용해 유닛을 랜덤으로 소환합니다.

재미 요소:

- 원하는 유닛이 바로 나오지 않는 운 요소
- 낮은 등급 유닛이라도 조합과 배치에 따라 의미가 생김
- 희귀 유닛이 등장했을 때의 즉각적인 만족감
- 매 판 다른 흐름

우리 게임 적용 방향:

- `랜덤 호출` 또는 `지원 요청`이라는 표현 사용
- 디스코드 커뮤니티 감성에 맞춰 병사, 마스코트, 드론, 해커, 힐러 같은 오리지널 유닛 구성
- 특정 원작 캐릭터 또는 스킬을 연상시키는 이름은 사용하지 않음

### 2.2 웨이브 디펜스

적이 일정 경로를 따라 이동하고, 플레이어가 유닛으로 막는 구조입니다.

재미 요소:

- 시간이 지날수록 커지는 압박
- 초반 운과 후반 빌드의 결합
- 보스 웨이브에서 긴장감 상승
- 짧은 플레이 타임으로 반복 플레이 유도

우리 게임 적용 방향:

- 1판 5~10분 내외의 짧은 웹게임으로 설계
- 초반은 쉽고 빠르게, 후반은 조합과 성장의 영향이 커지도록 설계
- 디스코드 이벤트용으로 특정 기간 한정 웨이브 추가 가능

### 2.3 합성/승급

동일 조건의 유닛을 합쳐 더 높은 등급 또는 더 강한 유닛으로 변환합니다.

재미 요소:

- 낮은 등급 유닛도 버리지 않고 성장 재료로 활용
- 합성 결과에 다시 랜덤성이 붙어 긴장감 형성
- 필드 공간 관리와 성장 선택이 동시에 발생

우리 게임 적용 방향:

- 같은 등급 2개 또는 3개를 합쳐 상위 등급으로 변환
- 합성 결과는 같은 계열 확률 증가 또는 완전 랜덤 중 선택 가능
- 초반 MVP에서는 단순 규칙으로 시작

### 2.4 유닛 등급

일반, 희귀, 영웅, 전설, 신화 등 여러 등급을 둡니다.

재미 요소:

- 수집 욕구
- 성장 목표
- 희귀 등장 연출
- 랭킹 경쟁에서 빌드 차별화

우리 게임 적용 방향:

- 등급명은 원작과 겹치지 않도록 새로 정할 수 있음
- 예시: Basic, Rare, Epic, Prime, Mythic
- 또는 서버 감성에 맞춰 `일반`, `특수`, `에이스`, `전설`, `비밀병기` 등으로 구성

### 2.5 유물/패시브 성장

게임 외부에서 장기적으로 누적되는 강화 요소입니다.

재미 요소:

- 매일 접속 동기
- 계정 성장감
- 시즌/이벤트 보상과 연결
- 유저별 플레이 스타일 차이

우리 게임 적용 방향:

- 초기에는 단순 패시브 카드 10~20개로 시작
- 과금이 아니라 서버 활동, 시즌 보상, 이벤트 보상으로 획득
- 너무 강한 영구 성장으로 신규 유저 진입 장벽이 생기지 않도록 제한

### 2.6 펫/동료 시스템

메인 유닛과 별개로 보조 효과를 주는 장기 성장 시스템입니다.

재미 요소:

- 수집 및 꾸미기
- 플레이 스타일 보조
- 커뮤니티 이벤트 보상으로 적합

우리 게임 적용 방향:

- `펫`보다는 서버 감성에 맞춰 `마스코트`, `봇 친구`, `동료 드론` 같은 이름 사용 가능
- 초기 MVP에서는 구현하지 않고 문서와 DB 설계만 준비
- 추후 시즌 보상과 연결

### 2.7 모드 구분

일반, 어려움, 도전, 이벤트, 경쟁 등의 모드 구분이 있습니다.

재미 요소:

- 숙련자용 목표 제공
- 짧은 플레이와 장기 도전의 병행
- 시즌 이벤트 운영 가능

우리 게임 적용 방향:

- MVP: 기본 웨이브 모드 1개
- 1차 확장: 하드 모드, 주간 도전
- 2차 확장: 실시간 듀오 협동, 길드/서버 전체 레이드

## 3. 우리 프로젝트에 맞는 재구성 방향

참고 장르를 그대로 재현하는 것이 아니라, 디스코드 커뮤니티 서버 전용 웹게임으로 재구성합니다.

### 3.1 플레이 감성

원작형 모바일 게임 감성보다, 웹에서 가볍게 열고 친구들과 기록을 비교하는 커뮤니티 미니게임 감성을 우선합니다.

핵심 감성:

- 가볍다
- 빠르다
- 운빨이 강하다
- 웃긴 결과가 자주 나온다
- 서버 랭킹으로 자랑할 수 있다
- 디스코드에서 공유하기 쉽다

### 3.2 오버워치 커뮤니티와의 연결

오버워치 IP를 직접 사용하지 않습니다.

대신 다음처럼 커뮤니티 분위기만 반영합니다.

- 팀 조합 감성
- 딜러/탱커/서포터 역할 구분
- 궁극기 느낌의 필살기 시스템
- 하이라이트/캐리/트롤 같은 커뮤니티 용어 감성
- 도파민, 랜덤, 합성, 대박, 억까 같은 밈 감성

주의:

- 오버워치 영웅명, 스킬명, 로고, 이미지 사용 금지
- 블리자드 고유 IP와 혼동될 수 있는 요소 지양

### 3.3 게임명/세계관 재정의

가칭 세계관:

`도파민 서버를 침공하는 버그 몬스터를 막기 위해, 랜덤으로 호출되는 서버 수호대가 웨이브를 방어한다.`

예시 용어:

| 장르 요소 | 우리 프로젝트 용어 후보 |
|---|---|
| 영웅 | 수호대, 요원, 서버 가디언 |
| 유물 | 서버 유물, 패시브 칩, 강화 모듈 |
| 펫 | 마스코트, 봇 친구, 드론 |
| 몬스터 | 버그, 잡몹, 침입자, 오류체 |
| 보스 | 대형 버그, 서버 크래셔 |
| 소환 | 호출, 지원 요청, 랜덤 배치 |
| 합성 | 업링크, 동기화, 합체, 승급 |
| 골드 | 포인트, 에너지, 서버 코인 |

## 4. 금지 기준

다음은 개발 중 절대 피해야 합니다.

1. 참고 게임의 캐릭터 이름을 그대로 사용하지 않는다.
2. 참고 게임의 스킬 이름을 그대로 사용하지 않는다.
3. 참고 게임의 아이콘, UI, 이미지, 배치 형태를 그대로 재현하지 않는다.
4. 참고 게임의 수치표를 복사하지 않는다.
5. 참고 게임의 과금 구조를 복제하지 않는다.
6. 참고 문서의 텍스트를 그대로 서비스 문구에 사용하지 않는다.
7. 오버워치 공식 IP와 혼동될 수 있는 명칭, 이미지, 영웅명, 스킬명을 쓰지 않는다.

## 5. 도입할 시스템 우선순위

### MVP 필수

- 랜덤 유닛 호출
- 유닛 등급
- 유닛 합성
- 웨이브 진행
- 보스 웨이브
- 게임 결과 저장
- 개인/서버 랭킹

### 1차 확장

- 유물/패시브 카드
- 일일 미션
- 시즌 랭킹
- 관리자 이벤트 설정
- 유닛 도감

### 2차 확장

- 마스코트/펫
- 주간 도전 모드
- 실시간 듀오 협동
- 디스코드 역할 보상
- 서버 레이드

### 보류

- 실시간 PvP
- 복잡한 길드전
- 현금 결제
- 거래 시스템

## 6. 핵심 차별화 포인트

1. 디스코드 서버 멤버십 기반 게임이다.
2. 서버 랭킹과 디스코드 이벤트가 자연스럽게 연결된다.
3. 게임 결과를 채널에 공유할 수 있다.
4. 운영자가 이벤트/시즌/보상을 웹에서 관리할 수 있다.
5. 게임이 웹 기반이라 설치 없이 바로 즐길 수 있다.
6. 원작과 달리 커뮤니티 활동 보상이 게임 진행과 연결된다.

## 7. 설계 결론

이 프로젝트는 `운빨 협동 디펜스` 장르의 핵심 재미를 참고하되, 디스코드 커뮤니티용 웹게임으로 재설계합니다.

초기에는 게임성을 과하게 확장하지 말고, 다음 4가지를 완성하는 데 집중합니다.

1. 로그인
2. 게임 한 판
3. 결과 저장
4. 서버 랭킹

이 4개가 안정화되면 유물, 마스코트, 시즌, 협동 모드를 단계적으로 추가합니다.
